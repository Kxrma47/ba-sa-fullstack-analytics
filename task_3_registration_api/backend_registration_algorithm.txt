Задание 3.2. Backend-алгоритм регистрации пользователя
======================================================

Endpoint
--------

POST /api/v1/users/register

Пошаговый алгоритм
------------------

1. Получить HTTP-запрос от кнопки Register.
2. Создать идентификатор запроса для логирования и трассировки ошибок.
3. Проверить, что метод запроса равен POST.
4. Проверить, что Content-Type равен application/json.
5. Прочитать тело запроса.
6. Разобрать JSON.
7. Если JSON некорректный, вернуть 400 Bad Request с errorCode INVALID_JSON.
8. Проверить обязательные поля:

   firstName
   lastName
   username
   password
   captchaToken

9. Если хотя бы одно обязательное поле отсутствует или пустое, вернуть 400 Bad Request с ошибками по полям.
10. Удалить начальные и конечные пробелы из firstName, lastName и username.
11. Нормализовать username для проверки уникальности: привести к нижнему регистру.
12. Проверить firstName:

    обязательно
    от 1 до 50 символов
    разрешены буквы, пробелы, дефис и апостроф

13. Проверить lastName:

    обязательно
    от 1 до 50 символов
    разрешены буквы, пробелы, дефис и апостроф

14. Проверить username:

    обязательно
    от 3 до 30 символов
    начинается с буквы
    может содержать латинские буквы, цифры, точку, нижнее подчеркивание и дефис
    уникален без учета регистра

15. Проверить password:

    обязательно
    от 8 до 128 символов
    минимум одна заглавная латинская буква
    минимум одна строчная латинская буква
    минимум одна цифра
    минимум один не буквенно-цифровой символ
    не содержит username

16. Проверить captchaToken:

    обязательно
    непустая строка
    сервер должен проверить токен у CAPTCHA-провайдера

17. Если CAPTCHA отсутствует или не подтверждена, вернуть 422 Unprocessable Entity с errorCode CAPTCHA_INVALID.
18. Начать транзакцию базы данных.
19. Проверить, существует ли пользователь с нормализованным username.
20. Если username уже существует, откатить транзакцию и вернуть 409 Conflict с errorCode USER_EXISTS.
21. Хешировать пароль надежным односторонним алгоритмом.
22. Создать запись пользователя со статусом active.
23. Сохранить created_at и updated_at.
24. Зафиксировать транзакцию.
25. Отправить событие подтверждения или email, если продукт требует подтверждения.
26. Вернуть 201 Created с данными созданного пользователя и сообщением об успехе.

Защита на уровне базы данных
----------------------------

В таблице users должен быть уникальный индекс по normalized_username. Backend должен корректно обрабатывать ошибку duplicate key, потому что две заявки на регистрацию могут прийти одновременно.

Требования безопасности
-----------------------

1. Не возвращать password или password hash ни в одном ответе.
2. Не записывать password или CAPTCHA token в прикладные логи.
3. Использовать HTTPS в production.
4. Применять rate limiting по IP-адресу и username.
5. Использовать транзакцию для проверки уникальности и создания пользователя.
6. Возвращать стабильные клиентские коды ошибок, но не раскрывать stack trace.
7. Хранить пароль только как salted one-way hash.

Условие успешного ответа
------------------------

Backend может вернуть 201 Created только после успешной валидации всех полей, подтверждения CAPTCHA, проверки свободного username, хеширования пароля и фиксации записи пользователя в базе данных.
